SSH and SSH Keys

A fundamental computer-usage topic for CS students. SSH connects directly to pushing to GitHub, and teaches the core idea of public-key cryptography through something students actually use.

What SSH is

  • SSH ("Secure Shell") is a protocol for connecting to another machine, or authenticating to a service, over an encrypted connection.
  • Two everyday uses for a beginner:
    • Logging into a remote computer/server and using its terminal.
    • Authenticating to GitHub so you can push/pull without typing a password.

The core idea: key pairs

  • SSH authentication uses a key pair: a private key and a public key.
  • The private key stays on your machine and is never shared — treat it like the physical key to your house.
  • The public key can be shared freely — think of it as a lock you hand out. Anyone can lock with it, but only your private key can unlock.
  • This is the same public/private idea behind a lot of modern security; SSH is a concrete first encounter with it.

Generating a key pair

  • Command: ssh-keygen -t ed25519 -C "your_email@example.com"
    • ed25519 is a modern, recommended key type.
    • -C adds a comment (usually your email) to label the key.
  • By default keys land in ~/.ssh/:
    • id_ed25519 — the private key (keep secret).
    • id_ed25519.pub — the public key (safe to share).
  • The passphrase prompt: an optional extra layer that encrypts the private key on disk, so a stolen laptop doesn't hand over the key.

Connecting SSH to GitHub

  1. Generate a key pair (above).
  2. Copy the public key (cat ~/.ssh/id_ed25519.pub).
  3. Paste it into GitHub → Settings → SSH and GPG keys → New SSH key.
  4. Use the SSH remote URL (git@github.com:user/repo.git), not the HTTPS one.
  5. Test with ssh -T git@github.com — GitHub greets you by username on success.
  • Payoff: git push / git pull now work with no password prompt.

The ~/.ssh/ directory

  • id_ed25519, id_ed25519.pub — your key pair.
  • known_hosts — servers you've connected to before (basis of the "authenticity of host can't be established" prompt on a first connection).
  • config — optional shortcuts and per-host settings.

Safety habits to teach

  • Never share or commit a private key. It should never appear in a repo — a natural tie-in to .gitignore.
  • Public key = shareable; private key = secret. Reinforce which is which.
  • Use a passphrase on the private key.
  • If a private key is ever exposed, remove it from GitHub and generate a new pair.

Common student pitfalls

  • Confusing which key to upload (it's always the .pub one).
  • Pasting the key into the wrong GitHub settings page.
  • Using the HTTPS remote URL and wondering why SSH keys aren't being used.
  • Permissions errors — SSH refuses to use keys that are too widely readable.
