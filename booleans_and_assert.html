Booleans and the Assert Statement

Comparing Values

Python can compare two values and tell you whether the comparison is true.

print(3 < 5)
print(3 > 5)
print(3 == 5)

True
False
False

The symbols < and > mean what you think they mean. The symbol == checks whether two values are equal. Notice that it's two equals signs, not one. A single = is for assignment (see Arithmetic and Evaluation). A double == is for comparison. Mixing these up is one of the most common mistakes a new programmer makes.

There are a few more comparison operators.

print(3 <= 3)
print(3 >= 5)
print(3 != 5)

True
False
True

The operator <= means less than or equal to, and >= means greater than or equal to. The operator != means not equal to.

Identity: is and is not

There is another way to compare two things, and it asks a different question.

n = 500
a = n * 2
b = 1000
print(a == b)
print(a is b)

True
False

Both a and b hold the value 1000, so a == b is True. But a is b is False. The values are equal, but they are not the same object.

== asks, "do these have the same value?" is asks a stricter question: "are these literally the same object, stored in the same place?" Python computed a fresh from n * 2, and it built a brand new 1000 to hold the result. The b = 1000 on the next line made a separate 1000, a different object that happens to hold an equal value.

a = 1000
c = a
print(a is c)

True

Now a is c is True, because c doesn't hold a copy. It refers to the exact same object that a refers to.

is has an opposite, is not, which works the way you'd guess.

n = 500 
a = 1000
b = n * 2
c = a
print(a is not b)
print(a is not c)

True
False

Key Idea: == compares values. is compares identity, whether two names point to the exact same object in memory.

In practice, is shows up most often with a special value called None, which represents "no value at all."

x = None
print(x is None)
print(x is not None)

True
False

The convention in Python is to check for None with is, not ==. You'll see is None and is not None constantly once you start writing real programs.

The Boolean Type

Each of those comparisons produced a value, either True or False. Let's look at the type of that value.

x = 3 < 5
print(x)
print(type(x))

True
<class 'bool'>

The class is called bool, short for Boolean, named after the mathematician George Boole. A Boolean is a value that can only be True or False. There is no third option. Notice the capitalization: True and False, not true and false. Python is picky about this.

Key Idea: A Boolean is a value, just like a number or a string. It can be stored in a variable, printed, and passed around like anything else.

Arithmetic with Booleans

Here's something strange.

print(True + True)

2

Some people expect an error here. Instead, we get 2.

print(True + True == 2)
print(int(True))
print(int(False))

True
1
0

Under the hood, True behaves like 1 and False behaves like 0. This is because bool is secretly a subclass of int in Python. Every Boolean is also, in some sense, a number. This is a good example of a mystery worth chasing rather than a mistake to fear.

Combining Conditions: and, or, not

You can combine Boolean values using the keywords and, or, and not.

print(True and False)
print(True or False)
print(not True)

False
True
False

The word and gives True only when both sides are True. The word or gives True when at least one side is True. The word not flips a Boolean to its opposite.

These combine naturally with comparisons.

age = 15
print(age >= 13 and age <= 19)

True

This expression checks whether age is a teenager's age. Each side of and is its own Boolean expression, evaluated first, then combined. This is just substitution again (see Arithmetic and Evaluation), the same idea you've already seen with arithmetic.

The assert Statement

Once you can write an expression that evaluates to True or False, you can use it to make Python check your work. The assert statement takes a Boolean expression. If the expression is True, nothing happens. If it's False, the program crashes.

x = 2 + 2
assert x == 4
print("Made it past the assertion.")

Made it past the assertion.

Nothing dramatic happened, because the assertion was True. Let's break it on purpose.

x = 2 + 2
assert x == 5
print("Made it past the assertion.")

Traceback (most recent call last):
  File "<stdin>", line 2, in <module>
AssertionError

The program stopped with an AssertionError, and the print line never ran. This is the point of assert. You are telling Python, "I believe this is true, and if it isn't, stop everything immediately."

This is also your first tool for testing code. Testing means checking that the code you wrote actually produces the output you expected. Suppose you write some arithmetic to compute the area of a rectangle, and you already know what the answer should be.

width = 4
height = 5
area = width * height
assert area == 20
print("The area calculation passed its test.")

The area calculation passed its test.

You didn't just compute area and hope it was right. You wrote down your expectation, 20, and had Python check the real output against it. If a typo or a wrong formula ever changed area, the assert would catch it immediately instead of letting a wrong answer quietly slip through. This habit, computing something and then asserting that it matches what you expected, is the same basic idea behind every testing tool you'll ever use, no matter how fancy.

Key Idea: An assertion is a claim about your program that you are willing to bet on. If the bet is wrong, you want to know right away, not many lines later when the mistake has already caused more damage.

Assert with a Message

You can add a second part to an assert statement, separated by a comma, to explain what went wrong.

x = -5
assert x >= 0, "x should never be negative"

Traceback (most recent call last):
  File "<stdin>", line 2, in <module>
AssertionError: x should never be negative

The message shows up alongside the AssertionError, which is much friendlier than a bare crash. When you write your own code later on, you'll find that a well-placed assert with a good message saves you a lot of guessing when something goes wrong.
