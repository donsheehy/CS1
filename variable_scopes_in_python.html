Variable Scopes in Python

A variable's scope is the region of your program where that variable's name is visible and can be used. Understanding scope answers questions like: "Why can't I use this variable outside the function?" and "Why didn't my function change my variable?" This guide walks through the rules Python uses.

The core idea

When you use a name (like x), Python has to figure out which x you mean. It searches through a series of scopes in a fixed order until it finds the name. When you assign to a name, Python decides which scope the name belongs to based on where the assignment happens.

The LEGB rule

To look up a name, Python searches four scopes in order:

  1. L — Local: names defined inside the current function.
  2. E — Enclosing: names in any function that encloses the current one.
  3. G — Global: names defined at the top level of the module (file).
  4. B — Built-in: names Python provides automatically, like print, len, range.

Python stops at the first match. If the name isn't found in any of the four, you get a NameError.

x = "global"          # global scope

def outer():
    x = "enclosing"   # enclosing scope (relative to inner)

    def inner():
        x = "local"   # local scope
        print(x)      # finds local first -> "local"

    inner()

outer()

If you remove the local x, inner finds the enclosing one; remove that too, and it finds the global one. Python always walks outward, never inward.

Local scope

A variable assigned inside a function is local to that function. It exists only while the function runs and is invisible outside it.

def greet():
    message = "hello"   # local
    print(message)

greet()
print(message)          # NameError: name 'message' is not defined

Function parameters are local variables too. Each call gets its own fresh set of locals, which is why functions don't accidentally share data between calls.

Global scope

Names defined at the top level of a file are global (module-level). They can be read from inside functions without any special declaration:

count = 0

def show():
    print(count)   # reading a global is fine -> 0

show()

But assigning to that name inside a function creates a new local variable instead of changing the global one:

count = 0

def increment():
    count = count + 1   # error: see below

increment()

This raises UnboundLocalError. Because increment assigns to count, Python treats count as local throughout the whole function — so count + 1 tries to read a local that hasn't been given a value yet.

The global keyword

To actually reassign a global from inside a function, declare it global:

count = 0

def increment():
    global count
    count = count + 1

increment()
increment()
print(count)   # 2

Use global sparingly. Functions that quietly modify globals are harder to reason about and test. Prefer returning a value and letting the caller decide what to do with it:

def increment(count):
    return count + 1

count = 0
count = increment(count)   # explicit and easy to follow

Enclosing scope and nonlocal

When you nest a function inside another, the inner function can read variables from the outer (enclosing) function:

def outer():
    tally = 0

    def add():
        print(tally)   # reads the enclosing tally -> 0

    add()

outer()

Just like with globals, assigning to tally inside add would make a new local. To reassign the enclosing variable, use nonlocal:

def make_counter():
    count = 0

    def increment():
        nonlocal count
        count += 1
        return count

    return increment

counter = make_counter()
print(counter())   # 1
print(counter())   # 2
print(counter())   # 3

nonlocal tells Python to bind to the nearest enclosing function's variable, not to create a local one and not to reach all the way out to the global scope.

Built-in scope

The outermost scope holds names Python provides for you: print, len, sum, range, list, dict, and many more. You never declare these, but they're findable from anywhere.

A common bug is shadowing a built-in by using its name as a variable:

list = [1, 2, 3]        # 'list' now refers to your list, not the built-in
new = list(range(5))    # TypeError: 'list' object is not callable

The fix is simply to avoid built-in names for your own variables. Good alternatives: items, values, numbers, my_list.

Scope and blocks: an important non-rule

Unlike many languages (C, Java, JavaScript with let), Python does not create a new scope for if, for, while, or with blocks. Variables assigned inside these blocks live in the enclosing function or module, not the block:

if True:
    y = 10

print(y)   # 10 — y is still visible

for i in range(3):
    pass

print(i)   # 2 — the loop variable survives after the loop

Only functions (and classes, and comprehensions) introduce new scopes.

Comprehensions have their own scope

Every comprehension (list, set, dict) and generator expression runs in its own function-like scope. Unlike a plain for loop, the loop variable does not leak into the surrounding code:

squares = [i * i for i in range(3)]
print(i)   # NameError: name 'i' is not defined

Because names assigned inside a comprehension are local to it, they can't clobber your outer variables. The comprehension can still read enclosing and global names through the usual LEGB search:

factor = 10
scaled = [n * factor for n in range(3)]   # reads 'factor' from outside -> [0, 10, 20]

The mechanism is that Python effectively rewrites a comprehension into a hidden function. This:

result = [f(x) for x in some_iterable if cond(x)]

behaves roughly like:

def _comp(it):
    out = []
    for x in it:
        if cond(x):
            out.append(f(x))
    return out
result = _comp(iter(some_iterable))   # outermost iterable evaluated outside

One subtlety: the iterable of the outermost for is evaluated in the enclosing scope, but everything else runs inside the comprehension's own scope. That's why the loop variable is contained — it's a local of that hidden function.

A gotcha appears inside a class body: a comprehension can't see the class's other attributes, because class scopes aren't part of the enclosing-function chain.

class C:
    values = [1, 2, 3]
    factor = 10
    doubled = [v * 2 for v in values] 

print("This works: 'values' is the outermost iterable")

This works: 'values' is the outermost iterable

class C:
    values = [1, 2, 3]
    pairs = [v * n for v in values for n in values]  # NameError: the *second*
    #     'values' is not the outermost iterable, so it runs in the comprehension's
    #     own scope and can't see the class attribute

  File "<stdin>", line 1, in <module>
  File "<stdin>", line 3, in C
NameError: name 'values' is not defined
class C:
    values = [1, 2, 3]
    factor = 10
    scaled = [v * factor for v in values]  # NameError: 'factor' is a class attribute

  File "<stdin>", line 1, in <module>
  File "<stdin>", line 4, in C
NameError: name 'factor' is not defined

Only the outermost iterable gets the special treatment of being evaluated in the enclosing (here, class) scope. Every other name in the comprehension — including a second for's iterable — is resolved inside the comprehension's own scope.

Quick reference

Situation What to use
Read a global inside a function Nothing special — just read it
Reassign a global inside a function global name
Read an enclosing variable in a nested function Nothing special — just read it
Reassign an enclosing variable in a nested function nonlocal name
Avoid a NameError Make sure the name is assigned before use in a reachable scope
Avoid an UnboundLocalError Don't assign to a name you also want to read as a global/enclosing, unless you declare it

Common pitfalls, summarized

Assigning to a name anywhere in a function makes it local for the entire function, even on lines before the assignment. This is the usual cause of UnboundLocalError.

Reading is permissive (LEGB search), but writing is strict (defaults to local). The global and nonlocal keywords are the only ways to change where a write lands.

Shadowing built-ins like list, dict, sum, id, or type doesn't error immediately — it quietly replaces them and causes confusing failures later.

Loop variables and variables set inside if/for blocks persist after the block, because blocks aren't scopes.

Mental model to remember

To look up a name, search outward: L → E → G → B. To assign a name, Python writes to the local scope by default; use global to write to the module scope and nonlocal to write to an enclosing function's scope.
